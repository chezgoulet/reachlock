# S30 — Agent Tooling

**Spec:** §13 (development principles), §10 (CLI tools) · **Wave 7 (tooling) ·
Depends on:** — (standalone, reads existing codebase)

## Outcome

Agentic coding tools (Claude Code, opencode, cursor, etc.) working on this
project get fast, deterministic feedback instead of re-discovering the same
information through file exploration. Three deliverables: an agent CI gate
that catches iron-rule violations in under 10 seconds, auto-generated agent
context files that answer structural questions without reading code, and a
`reachlock codex` CLI that agents invoke directly instead of grepping.

This is tooling FOR tooling. It makes every future agent session faster and
fewer tokens burned on exploration = more tokens spent on implementation.

## Context

- The project already has excellent human-facing documentation: sprint
  briefs, AGENTS.md, CLAUDE.md with path-scoped rules, gotcha ledger. What
  it lacks is MACHINE-ANSWERABLE questions — things an agent can resolve
  without reading 10 files.
- Agents working on this project spend 30-40% of their context window on
  exploration: finding patterns, checking crate boundaries, locating types,
  verifying that a new module follows the right conventions. This sprint
  eliminates that overhead.
- The existing `reachlock-cli` already has `gen`, `determinism`, and `content`
  subcommands. `codex` is a new subcommand family that lives alongside them.
- The agent CI gate is additive to `make check` — it runs FIRST because it
  catches agent-specific mistakes in seconds. `make check` still runs after
  and is still the gate. Think of it as a fast pretest.
- The auto-generated context files live in `docs/agents/` and are imported
  by CLAUDE.md via `@docs/agents/AGENT-INDEX.md` at session start. They're
  regenerated by `reachlock codex update` after structural changes (new
  modules, new types, sprint merges).

## Freeze first

### Agent check exit codes and output format

```
reachlock check agent [--json]
  exit 0: all checks pass
  exit 1: one or more checks failed (human-readable report)
  exit 2: internal error (unable to run checks)

JSON output (--json):
  { "passed": bool, "checks": [ { "name": str, "passed": bool, "detail": str } ] }
```

The JSON format is machine-parseable for CI and agent consumption. The human
format is for the developer reading the output in a terminal.

### Codex subcommand contract

```
reachlock codex <subcommand> [args]
  pattern   --type <kind>     print pattern template
  brief     <sprint-id>       print sprint context
  types     --crate <name>    list public types
  deps      --module <path>   print dependency graph
  update                      regenerate agent context files
  diff      --since <ref>     iron-rule violation scan on a diff
```

All commands output to stdout. All commands exit 0 on success, non-zero on
error with a message to stderr. Output is markdown (human) unless `--json`
is specified (machine).

### Context file format

`docs/agents/AGENT-INDEX.md` — auto-generated, header warns "DO NOT EDIT".
`docs/agents/AGENT-PATTERNS.md` — hand-curated, updated when new patterns
emerge. Both are valid markdown importable by CLAUDE.md.

## Deliverables

### 1. Agent CI gate (`reachlock-cli/src/agent_check.rs`)

A new subcommand: `reachlock check agent`. Runs a battery of fast checks
targeting the specific mistakes agents make. Each check is a standalone
function that returns `Pass` or `Fail(reason)`. All checks run independently
(no cascading failures). Target: exit 0/1 in under 10 seconds.

**Check suite:**

| Check | Rule | Implementation |
|---|---|---|
| `core-purity` | `reachlock-core/Cargo.toml` has zero rendering/IO deps. Bevy, reqwest, tokio, wgpu, rapier = fail. | Parse Cargo.toml (no `cargo` subprocess — fast). Allow-list of already-approved deps (serde, serde_json, sha2, ron). New dep not on the allow-list → fail with message "not proven wasm-safe; justify in PR". |
| `determinism-coverage` | Every `pub fn` in `core/src/generator/` has a corresponding golden key in `core/src/determinism.rs`. | Parse Rust source with `syn` (dev-dependency only). Extract all `pub fn` names in `generator/`. Grep for those names in `determinism.rs`. Missing → fail with the function name. |
| `no-floats-in-gameplay` | No `f32`/`f64`/`Float` in `reachlock-core/src/` outside allowed visual-only paths (`util/color.rs`, `util/trig.rs`). | Regex scan for `f32`, `f64`, `Float`, `float` in `.rs` files under `core/src/`. Subtract allowed files. Any hit → fail with file:line. |
| `wire-shape-pinned` | No `ClientMessage`/`ServerMessage` variant changed without the wire-shape test being updated. | Check if `network/messages.rs` was modified. If yes, verify the wire-shape test file has a change in the same commit or the test still passes. (For the fast gate: just check if messages.rs is dirty and the test file is NOT dirty → warn; `make check` catches the actual failure.) |
| `gotcha-scan` | Scan the staged diff for known gotcha patterns. | Regex patterns: `bevy::render::mesh::` (→ `bevy::mesh::`), `RapierPhysicsPlugin<NoUserData>` (→ `<()>`), `Timer.finished` (→ `is_finished`), `r#"` on lines containing `"#` (→ `r##"`), seed literal > 2^53, `f32`/`f64` type annotation in core. |
| `sprint-branch` | Branch name follows `sprint-v2/sXX-name`. If not on a sprint branch (e.g. `testing`, `main`), skip. If on a sprint branch, verify `docs/sprints/SXX-*.md` exists. | `git branch --show-current` → parse sprint ID → glob for matching brief. Brief missing → fail. |
| `module-registration` | New `pub mod X` in `lib.rs` or `mod.rs` → the corresponding file exists and has content. | Parse `lib.rs` and each crate's `mod.rs` for module declarations. For each, verify the file exists. Missing file → fail. |

**Tests:**
- `cargo test -p reachlock-cli agent_check::` — each check has a unit test
  with a known-bad fixture (e.g., a fake diff containing `f32` in core)
  that produces the expected failure.
- Integration: run `reachlock check agent` on a clean working tree → exit 0.
  Introduce a known violation → exit 1 with the check name in output.

**`docs/agents/AGENT-PATTERNS.md`** — hand-curated, updated when new patterns
are discovered by agents working on the project:

```markdown
# ReachLock Agent Patterns — hand-curated, add as you learn

## Adding a new game mode / panel
1. Add variant to `ActivePanel` (interaction.rs:88) or `GameMode` (states.rs:48)
2. Create `systems/my_panel.rs` with a Resource + keyboard system + HUD render fn
3. In main.rs: init the resource, add systems with run condition, add HUD update
4. Reference: `systems/market.rs` (ActivePanel keyboard pattern), `systems/onboard.rs`

## Adding a new generator
1. Create `core/src/generator/my_gen.rs` — pure fn, no IO, no floats in gameplay values
2. Add `pub mod my_gen;` to `core/src/generator/mod.rs`
3. Add golden entry to `core/src/determinism.rs`, bump manifest version
4. Commit message MUST mention "manifest vN→vM"

## Adding a new server endpoint
1. Add route to `ws/mod.rs` router()
2. Handler in `ws/handler.rs` or new `services/` file
3. If new message variant: update `network/messages.rs` AND the wire-shape test
4. New store behind a trait: memory default, Postgres under `postgres` feature

## Adding a new ECS system
1. Create `systems/my_system.rs` with pub fns
2. Add `pub mod my_system;` to `systems/mod.rs`
3. In main.rs: register with a run condition (in_spaceflight / in_any_interior / space_live / in_state)
4. If new component/resource: define in the system file or in core if it's shared
5. Tag scene entities with ModeScope if they should despawn on mode exit

## Adding a new Bevy resource
1. Define the struct (derive Resource, Default if possible)
2. In main.rs: .init_resource::<MyResource>() in the init block
3. Access in systems: `my_res: Res<MyResource>` or `ResMut<MyResource>`

## Working with determinism
1. If you touch a generator OR add one: extend determinism.rs
2. Manifest version goes up by 1 for each sprint's additions
3. Call out the version change in the commit message
4. `cargo test -p reachlock-core determinism` verifies goldens locally

## Common imports (keep these in muscle memory)
- core types: `use reachlock_core::{generator::*, util::rng::*, ...}`
- bevy mesh: `use bevy::mesh::Mesh` (NOT bevy::render::mesh)
- fixed point: `use reachlock_core::util::rng::Fixed` (1/1024 scale)
- seeds: `use reachlock_core::seed::{Seed, SystemId}` (seed ≤ 2^53)
```

### 3. `reachlock codex` CLI (`reachlock-cli/src/codex.rs`)

New subcommand family. Each command produces machine-parseable output that
answers a specific structural question an agent would ask.

**`reachlock codex pattern --type <kind>`**

Maps a task to the files, types, and patterns involved. Reads `AGENT-PATTERNS.md`
for curated entries; falls back to AST inspection for type location.

```
$ reachlock codex pattern --type add-game-mode
# Game Mode Addition Pattern
## Files to create
- reachlock-client/src/systems/my_mode.rs (new)
## Files to modify
- reachlock-client/src/states.rs:48 (GameMode enum)
- reachlock-client/src/main.rs:184 (OnEnter registration)
- reachlock-client/src/main.rs:31 (run condition)
## Patterns to follow
- systems/market.rs: ActivePanel + keyboard resource + HUD text render
- systems/onboard.rs: console panel variant
## Key types
- GameMode (states.rs:48)
- ActivePanel (interaction.rs:88)
- ModeScope (states.rs:76)
```

**`reachlock codex brief <sprint-id>`**

Summarizes a sprint brief, cross-referencing with the codebase to answer
"what's already there vs. what do I need to build".

```
$ reachlock codex brief S19
# S19 — Space Combat
## Dependencies
- S05 (items): merged ✓ — use ItemStats, ItemSeed, StatKey
- S09 (flight): merged ✓ — use HullHandling
## Types to define (from brief: Freeze first)
- BehaviorState: core/src/combat/behavior.rs (new)
- Intent: core/src/combat/behavior.rs (new)
- CombatVessel: core/src/combat/damage.rs (new)
## Existing types to reuse
- HullHandling → core/src/generator/hull.rs:100
- SeededRng → core/src/util/rng.rs:33
- Fixed → core/src/util/rng.rs:10
## Files to create
- core/src/combat/mod.rs, behavior.rs, damage.rs, encounter.rs
- client/src/systems/combat.rs
## Files to modify
- core/src/lib.rs (add pub mod combat)
- core/src/determinism.rs (add golden entries, bump manifest)
- client/src/main.rs (register resources + systems)
- client/src/systems/mod.rs (add pub mod combat)
## Gotchas from brief
- Combat math in core (integers), presentation in client (floats)
- Enemy Intents cap turn/thrust by THEIR HullHandling
- rapier collision groups: separate player/enemy/projectile from start
```

**`reachlock codex types --crate <name>`**

Prints `AGENT-TYPES.md` filtered by crate, or the full index if no filter.

**`reachlock codex deps --module <path>`**

Prints what a module imports and what imports it. Uses `syn` to parse the
AST and build a dependency graph.

```
$ reachlock codex deps --module core::generator::hull
# Dependencies of reachlock-core::generator::hull
## Imports
- core::util::rng::{Fixed, SeededRng, FixedVec2}
- core::generator::GeneratedMesh
## Imported by
- core::editor::exterior (uses HullClass, HullHandling)
- core::determinism (golden entry: hull_generation)
- core::generator::ship (uses HullClass for ship layout bounds)
- client::systems::ship (uses HullHandling for flight physics)
- client::systems::onboard (uses HullClass for interior grid size)
```

**`reachlock codex update`**

Regenerates `docs/agents/AGENT-INDEX.md` and `docs/agents/AGENT-TYPES.md`
from the current codebase and git state. Does NOT overwrite `AGENT-PATTERNS.md`
(that's hand-curated). Should be run after every sprint merge or major
structural change. Add to the PR checklist in CLAUDE.md.

**`reachlock codex diff --since <ref>`**

Scans the diff between HEAD and a reference for iron-rule violations. Same
checks as the agent gate but operates on a commit range rather than the
working tree. Used by CI for PR validation and by agents before committing.

## Acceptance gates

```
reachlock check agent                        # exit 0 on clean tree
reachlock check agent --json | jq .passed    # true
reachlock codex pattern --type add-generator # prints valid markdown, known pattern
reachlock codex brief S19                    # prints valid markdown, all sections
reachlock codex update                       # regenerates AGENT-INDEX.md + AGENT-TYPES.md
cargo test -p reachlock-cli agent_check:: codex::
make check
```

Manual: introduce a `f32` in `reachlock-core/src/sim.rs` → `reachlock check agent`
exits 1 with "no-floats-in-gameplay: sim.rs:42". Fix it → exits 0. Run
`reachlock codex brief S17` → output includes files that will be created.

## Non-goals

- Full semantic code search (embeddings, vector DB). The codebase is ~36K
  lines — AST parsing + grep is sufficient and deterministic.
- AI-generated sprint briefs. Briefs are design documents. The agent can't
  infer design intent from code.
- Automated PR review agent. Too many false positives on a game codebase.
  The agent gate catches the deterministic violations; humans review design.
- IDE integration (LSP, VS Code extension). The CLI is the interface. Agents
  and humans both use it from the terminal.
- `reachlock codex check-wasm` — `cargo check --target wasm32-unknown-unknown`
  already exists in `make check`. Not worth a separate subcommand.
- Live-updating agent context on file save. `reachlock codex update` is
  manual/CI-triggered. File watchers are complexity without proportional
  benefit.

## Gotchas

- `syn` parsing is a dev-dependency only. The agent check and codex commands
  use `syn` at build time, not at runtime. The binary doesn't carry a Rust
  parser — it reads pre-parsed data or uses fast regex checks.
- The `determinism-coverage` check is a heuristic — it matches function names
  as substrings in `determinism.rs`. A false negative is possible if a
  generator function's name appears in the determinism file as part of a
  different string. This is acceptable — the check is a fast pretest, not a
  proof. `cargo test -p reachlock-core determinism` catches real misses.
- `AGENT-TYPES.md` can grow large as more types are added. Keep it
  module-scoped with a table of contents. If it exceeds 500 lines, consider
  splitting into per-crate files (`AGENT-TYPES-core.md`, etc.).
- The gotcha-scan check must run on the STAGED diff (or working tree if
  nothing is staged), not the full codebase. Scanning all of core for `f32`
  would produce thousands of false positives from the bridge layer and
  allowed paths.
- `reachlock codex brief <id>` reads the brief markdown and cross-references
  with the codebase. If the brief format changes, the parser must be
  updated. Pin the brief format with a schema test (reuse the content schema
  validation pattern from S01).
