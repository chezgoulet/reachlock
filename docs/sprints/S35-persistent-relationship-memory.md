# S35 — Persistent Relationship Memory

**Spec:** §15 (soul system), §18 (LLM agency model) · **Wave 8 (LLM gameplay) ·
Depends on:** S13 (souls), S15 (agency), S16 (dialogue)

## Outcome

The crew remembers. Not as a database — as a story. Faction leaders, station
NPCs, and the player's own crew carry the accumulated weight of every
interaction. When the LLM deliberates, it doesn't just see the current
situation. It sees the history. "Tove has served on this ship for 47 days.
She's been right in 8 of 11 crises. The captain overruled her twice. She
stopped volunteering opinions after the Veil." That context changes HOW the
LLM reasons, which changes WHAT the crew says, which IS the emergent story.

This is the memory that makes characters feel lived-in. Not scripted arcs.
Accumulated experience.

## Context

- NPCs already have soul files with mood/memory state (S13). This sprint
  extends that from "authored memory entries" to "accumulated interaction
  history compressed into an LLM-optimized context."
- Every contract deliberation produces a trace (S15). Every co-deliberation
  produces relationship deltas (S33). Every faction interaction updates
  reputation (S11). Every crisis produces a log entry. The raw data exists
  scattered across systems. This sprint compresses it into a form the LLM
  can use at deliberation time.
- The research surface: how do LLM characters maintain consistency across
  weeks of real time and hundreds of interactions? What compression
  strategies produce coherent character arcs? What gets lost?
- The player creates their own crew, backgrounds, and starter attributes.
  Relationship memory is between the player's own characters AND between
  characters and NPCs/factions. The memory system doesn't care who authored
  the soul — it tracks all relationships equally.

## Freeze first

### Memory store (`soul/memory.rs`)

```rust
pub struct RelationshipMemory {
    pub memory_id: String,                  // "player::tove" or "boris::alexander"
    pub participants: (String, String),     // sorted alphabetically
    pub first_interaction: u64,             // tick when they first met
    pub interaction_count: u64,
    pub conversation_count: u64,
    pub crisis_count: u64,                  // times they've been in crisis together
    pub significant_events: Vec<SignificantEvent>, // capped at last 50
    pub compressed_context: Option<String>, // LLM-ready summary, regenerated periodically
    pub compression_version: u32,           // increments when context is rebuilt
    pub last_compression_tick: u64,
    pub trust_trajectory: TrustTrajectory,   // how trust has changed over time
}

pub struct SignificantEvent {
    pub tick: u64,
    pub event_type: SignificantEventType,
    pub summary: String,                     // one-line, generated by a small LLM call
    pub weight: Fixed,                       // how much this matters to the relationship
    pub fading: bool,                        // old events fade; recent ones are crisp
}

pub enum SignificantEventType {
    // Positive
    SavedMyLife,
    FollowedMyAdvice,
    ShowedTrust,
    DefendedMe,
    SharedSuccess,
    // Negative
    OverruledMe,
    EndangeredMe,
    BrokeTrust,
    AbandonedMission,
    WasWrong { consequences: String },
    // Neutral
    FirstMet,
    Reunited,
    SharedSilence,
    ObservedFromAfar,
    // Player-custom
    PlayerNoted { note: String },
}

pub struct TrustTrajectory {
    pub points: Vec<(u64, Fixed)>,           // (tick, trust_value) — last 20 inflection points
    pub trend: TrustTrend,                   // Rising, Falling, Stable, Volatile
}

pub enum TrustTrend {
    Rising { rate: Fixed },
    Falling { rate: Fixed },
    Stable,
    Volatile { amplitude: Fixed },
}
```

### Context compression strategy (`soul/compression.rs`)

The raw memory is too large for every LLM call. Compression produces a dense
summary designed for the LLM's context window:

```rust
pub struct CompressedContext {
    pub relationship_summary: String,        // "47-day crewmate. Trust: high. Respect: medium.
                                             //  Tension: low. Last significant event: defended
                                             //  her repair call during the Veil crisis."
    pub trajectory_narrative: String,         // "Trust has been rising steadily since day 12.
                                             //  Recent volatility: captain overruled her
                                             //  3 times this week. She's been quieter."
    pub key_memories: Vec<String>,           // top 5 events by weight, one sentence each
    pub compression_metadata: CompressMeta,
}

pub struct CompressMeta {
    pub compressed_at_tick: u64,
    pub event_count_compressed: u64,
    pub compression_strategy: CompressionStrategy,
}

pub enum CompressionStrategy {
    Summarize,       // standard: summarize all events into narrative
    TopWeight,       // tight context: only the 5 highest-weight events
    TrajectoryOnly,  // minimal: just the trajectory + trust level
}
```

The compression strategy is selected per LLM call based on the available
context budget (the `max_tokens` parameter on the LLM call, already known).

## Deliverables

### 1. Memory accumulation (`soul/memory.rs`)

- [ ] Every deliberation (S15) that involves an NPC or crew member writes to
      their `RelationshipMemory` with the other participant. A Boris
      deliberation about the captain writes to the Boris↔Captain memory.
- [ ] `SignificantEvent` creation: after each deliberation, a small
      classification step determines the event type and weight. This can be
      heuristic (S15's `LlmOutcome` → event type mapping) or a small LLM
      call ("classify this deliberation into an event type"). Heuristic
      first; LLM classification is feature-flagged.
- [ ] Fading: events older than 20 game-hours decay in weight. An argument
      last month is background texture; an argument today is sharp. The
      fading curve is linear from weight → 0 over 60 game-hours.
      `SignificantEvent.fading = true` when decay has started.
- [ ] Trust trajectory: after each event, append the new trust value to the
      trajectory points. If the trajectory has more than 20 points, collapse
      the oldest 10 into a single average point. Compute `TrustTrend` from
      the last 10 points via linear regression (simple integer math — no
      floats in core).

### 2. Context compression (`soul/compression.rs`)

- [ ] `compress(RelationshipMemory, strategy) -> CompressedContext`: produces
      the LLM-ready summary. The summary is deterministic from the memory
      state — same memory, same compression strategy, same output text.
      This matters for offline/online parity (the same memory should produce
      the same deliberation context).
- [ ] Compression triggers: after every N significant events (default: 5),
      or when the compressed context is older than M ticks (default: 1000
      ticks = ~2 hours of play). Compresses the new events into the
      `CompressedContext` and increments `compression_version`.
- [ ] Strategy selection: the LLM call's available token budget determines
      which strategy to use. `Summarize` for calls with 200+ tokens of
      context budget. `TopWeight` for 100-200. `TrajectoryOnly` for <100.
      The contract's `max_tokens` field controls the budget; relationship
      context is charged against that budget.
- [ ] Deterministic compression: `compress` uses template-based text
      generation (not an LLM). The templates are authored strings with
      interpolation. "{{name}} has served for {{days}} days. Trust is {{trust_word}}.
      Last significant event: {{event_summary}}." This ensures offline parity.

### 3. LLM context injection (`agency.rs` expansion)

- [ ] During `LlmService::route()`, the `context_json` now includes a
      `relationship_context` field for each relevant relationship. The
      existing `DialogueContext` builder (S16) is extended to include
      the compressed relationship memory.
- [ ] The system prompt is extended with relationship guidance: "You are
      speaking as {{character}}. You have the following relationship with
      {{interlocutor}}: {{compressed_context}}." The existing system prompt
      assembly (S15, S16) already supports this structure — the memory is
      injected as additional template variables.
- [ ] Context budget management: relationship memory competes with situation
      context for the LLM's context window. The `DialogueContext` builder
      (S16) already handles size-bounded assembly — relationship memory
      is another source with the same bounds.

### 4. Player visibility

- [ ] Crew roster → relationship tab: shows the relationship state between
      the player and each crew member, and between crew pairs. Not raw
      numbers — narrative labels. "Trust: High. Respect: Growing. Tension:
      Low. 47 days together. Last crisis: they argued, she was right."
- [ ] NPC interaction: during dialogue (S16), the relationship memory is
      visible as a "History" section in the dialogue panel. "You've done
      8 jobs for Alexander. He trusts your competence but not your motives."
- [ ] Player-noted events: the player can add a manual `SignificantEvent`
      from the crew roster or after a notable interaction. "Boris saved
      Tove during the Kessel blockade." This event gets full weight
      (never fades) and is tagged `PlayerNoted`. The player's own
      narrative choices become part of the memory system.

### 5. Performance and storage

- [ ] Memory per relationship: capped at 200 significant events (oldest
      are collapsed into the compressed context). A crew of 7 has 21
      pairwise relationships. At ~200 bytes per event, total memory
      is <100KB. Fits in a save file.
- [ ] Compression is O(n) in event count — only processes new events since
      last compression. Never O(n²).
- [ ] Persistence: relationship memory is part of the save file (like
      inventory and ship config). Backed up to the server via the save
      backup endpoint (S23). Deterministic from game events — can be
      reconstructed from the event log if a save is corrupted.

### 6. Metrics collection

- [ ] `relationship_memory_events` table: compression frequency, event
      counts per relationship, trust trajectory classification distribution.
- [ ] Research questions: does trust trajectory (Rising/Falling/Stable)
      correlate with player retention? What event types produce the largest
      trust deltas? How often does compression change the LLM's perceived
      relationship quality vs raw event history?


Manual: play 3 sessions with the same crew → check crew roster relationship
tab → trust trajectory shows changes from session events → enter dialogue
with a faction NPC → their dialogue reflects accumulated history → add a
player note → note appears in the relationship history.

## Non-goals

- Full replay/reconstruction of past events (the event log is the source;
  memory is the compressed view. Replay belongs to S37's captain's log.)
- Relationship between players (PvP social features are post-S23 MMO layer;
  this sprint covers NPC↔player and crew↔crew.)
- Relationship-driven quest generation (the memory feeds dialogue, not quest
  logic. The content pipeline authors quests; memory makes them personal.)
- Server-side relationship computation (memory is local, backed up to server.
  Server doesn't compute relationships — clients do, deterministically.
  Same seed + same events = same memory.)

## Gotchas

- Compression must be deterministic from event history. If the compression
  function uses an LLM, offline mode must use a pinned model version or a
  template-based fallback that produces identical output. The iron rule:
  core doesn't call LLMs. Compression is deterministic text templates in
  core; the LLM sees the output, not the process.
- Trust values are fixed-point (Fixed, 1/1024 scale, per iron rule #2).
  Trust trajectory linear regression must use integer math. A simple
  average-of-differences is sufficient for trend detection — no need for
  real linear regression.
- Event type classification mapping from `LlmOutcome` → `SignificantEventType`
  must be defined in the sprint. Not every LLM outcome maps cleanly. Define
  the mapping table in the module docs and test it.
- Memory per relationship grows linearly with events. The cap at 200 events
  must be enforced at write time. When the cap is hit, the 10 oldest events
  are collapsed into a single "Early history" compressed event. This is
  lossy compression by design — old events fade into background texture.
- Player-noted events have `PlayerNoted` type and never fade. This could
  create an unbounded growth problem if a player adds hundreds of notes.
  Cap player notes at 50 per relationship. Warn when approaching the cap.
